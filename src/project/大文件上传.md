### 背景
1. 大文件是指传送大于100M的文件，容易遇到问题：1.前后端上传请求超时限制，一次性传输大小限制。 2.网络抖动等，失败后需要重新上传。为了避免上传速度很慢，传输中断等问题，需要使用切片上传的方式。
2. 课件支持上传音频、视频、PPT、PDF等类型的课件；上传的音视频课件大于100M的文件，容易遇到上传速度很慢，传输中断，等待一段时间后传输失败等问题等问题
### 解决方案
1. 将要上传的大文件用`Blob.prototype.slice`分成许多大小相同的小块，利用http的可并发性同时上传多个切片。
2. 所有小块上传完成后，服务器会把这些小块按照正确的顺序拼接起来，还原成完整的大文件。这样就从原本上传一个大文件变成并发多个小文件，减少了上传时间。
3. 优点：1）可以加快上传速度：可以同时上传多个小块，充分利用网络的带宽。2）减少上传失败的风险：上传过程中出现了问题，只需要重新上传出错的那个小块。
### 目前成熟的大文件上传方案
1. 目前社区已经存在一些成熟的大文件上传解决方案，如七牛SDK，腾讯云SDK等,也许并不需要我们手动去实现一个简陋的大文件上传库
2. 前端vue组件`vue-simple-uploader`是基于`simple-Uploader.js`封装的大文件上传组件
### 实现
1. 文件分片：将大文件分割为多个小块,以4M为单位分割
    - 前端上传大文件时使用 `Blob.prototype.slice(file.slice)`将文件切片，数量由文件的大小控制，如在项目中设置的是4M，一个100M的文件会被分为25个切片。
2. 唯一标识：根据文件内容生成唯一hash值(如MD5)使用`spark-md5`和`Web Worker`根据文件切片算出文件hash
    - 主线程用`postMessage`发送切片数组
    - 在 worker 线程中接受文件切片 ，利用`fileReader`读取每个切片的`ArrayBuffer`并不断传入 spark-md5 中
    - 全部完成后将最终的`hash`发送给主线程，主线程用`onmessage`接收`hash`
3. 文件上传：发送前调用接口检查hash是否之前上传过
    - 若后端已有该文件，则直接返回成功，实现秒传；若服务端记录一些的分片信息则返回前端，前端过滤掉已完成的部分再上传，实现断点续传。
    - 遍历需上传切⽚的数组，每个`chunk`都要通过`new FormData()的append`⽅法进⾏`chunk hash filename index`的填充，返回一个通过`request`请求包裹的数组
    - `Promise.all`并发上传需要上传的切片，所有切片上传成功后向服务器发送一个合并的请求把切片合并成最终文件。
4. 暂停与继续上传：使用`XMLHttpRequest`的`abort`方法，可以取消一个xhr请求的发送
    - 暂停：有一个`requestList`数组存储正在上传切片的`xhr`，点击暂停按钮时候，调用保存在`requestList`中 xhr的`abort`方法，即取消并清空所有正在上传的切片；
    - 续传：是上传前验证文件hash，后端将已上传的切片的切片名返回，前端跳过这些已经上传切片。
### 难点
1. 使用`web-worker`在`worker`线程计算hash
```js
// /public/hash.js
// 参数是一个 js 文件路径且不能跨域，所以我们单独创建一个 hash.js 文件放在 public 目录下，
// 另外在 worker 中也是不允许访问 dom 的，但它提供了importScripts 函数用于导入外部脚本，
self.importScripts("/spark-md5.min.js");// 导入脚本
// 生成文件 hash
self.onmessage = e => {
  const { fileChunkList } = e.data;
  const spark = new self.SparkMD5.ArrayBuffer();
  let percentage = 0;
  let count = 0;
  const loadNext = index => {
    // 
    const reader = new FileReader();
    reader.readAsArrayBuffer(fileChunkList[index].file);
    reader.onload = e => {
      count++;
      spark.append(e.target.result);
      if (count === fileChunkList.length) {
        self.postMessage({
          percentage: 100,
          hash: spark.end()
        });
        self.close();
      } else {
        percentage += 100 / fileChunkList.length;
        self.postMessage({
          percentage
        });
        // calculate recursively
        loadNext(count);
      }
    };
  };
  loadNext(0);
};
// 主页面发送信息，切片内容
calculateHash(fileChunkList) {
   return new Promise(resolve => {
      // 添加 worker 属性
      this.container.worker = new Worker("/hash.js");
      this.container.worker.postMessage({ fileChunkList });
      this.container.worker.onmessage = e => {
        const { percentage, hash } = e.data;
        this.hashPercentage = percentage;
        if (hash) {
          resolve(hash);
        }
      };
    });
  },
```
2. 如何实现秒传?
    - 秒传功能依赖于文件唯一标识hash,前端读取文件内容，计算其`hash`；
    - 上传前先调用后端`checkFile(hash)`接口，检验一下是否存在该hash文,若后端已有该文件，则直接返回成功，无需实际上传；
3. 后端如何合并分片？如何保证顺序和完整性？
    - 前端在上传完成后，调用`mergeChunks(fileHash, totalChunks)`；
    - 后端根据`fileHash + chunkIndex`排序分片；合并时校验每片大小和顺序是否正确；合并后进行hash校验，验证是否和前端一致。
4. 上传中断后如何恢复？是自动续传的吗？
    - 上传前计算文件`hash`；若中断重新打开页面或网络恢复后，自动读取上传状态；查询已上传分片列表，只上传剩余部分。
