### npm与yarn
1. npm早期
    - npm在安装依赖有层级嵌套问题,如包A有依赖B,那么B安装在A的`node_modules`文件夹中，B可能也有自己的依赖包C,那么会出现依赖递归或依赖重复下载问题。
    - 依赖包会出现重复,出现体积过大，内存浪费->目录层级过深,导致文件路径过长,会出现删除不掉的情况或不好调试的问题。
2. yarn
    - 出现是为了解决问题：npm还没有`package-lock.json`文件,安装的时候速度很慢,稳定性很差，下载速度慢。
    - 确定性: 通过yarn.lock；这个文件的依赖版本号不是固定的，结构非json结构，而是一种自定义的标记模式。
    - 扁平化：安装依赖为模块扁平化方式；将不同版本的依赖包，安装一定的策略归结为单个版本。
    - 网络性能更好：yarn采用了请求排队的理念,类似于并发池连接；
    - 采用缓存机制：实现了离线模式 
3. npm和yarn的区别是什么？
    - yarn是为了弥补npm下载速度慢、版本一致性的问题而产生的。在npm5.4.2版本后，因为竞争关系，他们都支持缓存、扁平化、包并行下载，速度都有明显提升。所有现在的npm和yarn都能选择，他们的性能、优化、缓存都差不多。
    - 不同的是npm是`package-lock.json`采用的是JSON的结构，依赖是固定的版本；Yarn是`yarn-lock.json`，结构是一种自定义的标记方式，子依赖的版本不是固定的版本。
4. 转换:`package-lock.json`与`yarn-lock.json`的转换有一个专门的工具`synp`。
5. 缺点:解决了嵌套方式的部分问题，但是引入了幽灵依赖的问题，并且同名的包只会提升一个版本的，其余的版本依然会复制多次。
### cnpm、npx和pnpm的区别
1. cnpm：淘宝npm镜像，是淘宝提供的一个npm镜像客户端，默认使用淘宝NPM镜像源`（registroy.npmmirror.com）`，属于国内的，速度更快。
2. npx：是工具包，它不需要全局安装，直接就能运行命令。如果命令的包未安装，那么会临时安装并执行
    - 会在当前目录下的`./node_modules/.bin`里去查找是否有可执行的命令
    - 没有找到的话再从全局里查找是否有安装对应的模块
    - 全局也没有的话就会自动下载对应的模块，用完就删，不会占用本地资源。
3. pnpm：是一个快速、节省磁盘的包管理工具，原理不是复制包而是通过link,用了软链接。
    - 什么是软链接？硬连接就是同一个文件的不同引用；而软链接是新建一个文件，文件内容指向另一个文件或目录的快捷方式。需要`npm link`来执行，改变的任何东西，源文件会同步进行变化。
    - `pnpm`解决了幽灵依赖，`yarn`与`npm`的扁平化处理，会导致明明没有在`dependencies`声明的依赖A，代码也可以直接`require`使用，这种不是显式依赖，某天依赖的依赖A直接不被使用，那么代码会有问题。
### 多包管理
1. 多包是指有多个包要管理，比如一个项目有多个业务组件，每个组件都是一个包，手动更新每一个包，（版本号或者具体哪个包）容易出错。`Lerna`、`Yarn Workspaces`可以实现多包管理。
2. 将代码拆分为多个独立的包，可以提高代码的可维护性、复用性和开发效率。 
3. 包（Package）通常是一个独立的模块或功能，包含自己的 package.json 文件。包可以是工具库、组件库或服务等。
#### Lerna
1. 适合包含多个独立包的大型项目。提供了版本管理、依赖管理、发布等功能。
2. 实践
    - 初始化项目：`npx lerna init`，初始化后会生成以下文件：
        ■ lerna.json：Lerna 配置文件。
        ■ package.json：根目录的package.json。
        ■ packages/：用于存放子包的目录。
    - 创建子包：在`packages`下创建两个子包，它们都有自己的`package.json`文件
    - 配置`lerna.json`文件
    - 安装依赖：`npx lerna bootstrap`，安装所有子包的依赖
    - 管理依赖：添加公共依赖，`npx lerna add lodash`；为utils包添加axios，`npx lerna add axios --scope=@my-monorepo/utils`
    - 运行脚本：运行所有包脚本test，`npx lerna run test`。运行utils包脚本test， `npx lerna run test --scope=@my-monorepo/utils`
    - 版本管理：`npx lerna version npx lerna publish from-package`
#### Yarn Workspaces
  a. Yarn 内置的多包管理功能。
  b. 目前最常见的`monorepo`解决方案是基于`lerna`和`yarn workspace`的`monorepo`工作流,用yarn来处理依赖问题，用lerna来处理发布问题。
  c. 实践
    ⅰ. 初始化项目：在 package.json 中配置 workspaces。
    ⅱ. 创建子包：在`packages`目录下创建子包。
    ⅲ. 安装依赖：`yarn install`添加所有子包的依赖。
    ⅳ. 添加依赖：使用 `yarn workspace`命令为特定包添加依赖。`yarn workspace @my-monorepo/utils add axios`
    ⅴ. 运行脚本：使用 `yarn workspaces run test`或`yarn workspace @my-monorepo/utils test`运行脚本。test是`package.json的"scripts": { "test": "echo 'Running utils tests'"}`
### 优雅的提交
git项目中，通常需要对提交信息（commit message）进行规范化和校验，以确保提交信息的格式符合团队或项目的标准
#### commitlint与husky: 检测提交信息的规范
1. `<type>(<scope>): <subject>`例子：`feat(login): add user login feature；fix(api): handle null response`
2. 方式一：使用`commit-msg`钩子校验提交信息，在git项目进入`.git/hooks`目录，创建`commit-msg`钩子文件，在`commit-msg`文件中编写校验逻辑（使用 Bash 和正则表达式），后面提交代码时，如果提交信息不符合规范，提交会被终止。
3. 方式二：使用`commitlint`工具自动化校验流程
  a. 安装commitlint：`npm install --save-dev @commitlint/cli @commitlint/config-conventional`
  b. 配置commitlint：`commitlint.config.js` 文件 `exports出extends: ['@commitlint/config-conventional']`,
  c. 配置git钩子：`npm install --save-dev husky`在在 `package.json `中添加配置，`"scripts": { "prepare": "husky install"},"husky": {"hooks": {"commit-msg": "commitlint -E HUSKY_GIT_PARAMS"}} `
  d. 命令初始化 `husky：npm run prepare`
  e. 后面提交不符合规范的提交信息会提示 `git commit -m "invalid message"`
#### eslint与lint-staged提交代码前对代码进行静态检查和格式化
1. ESLint负责代码质量检查，而 `lint-staged` 则只对暂存区（staged）的文件进行检查和修复。
2. 安装ESLint及其常用插件：`npm install --save-dev eslint eslint-plugin-react eslint-config-airbnb`
3. 配置ESLint: `npx eslint --init`初始化后可以修改`.eslintrc.js`文件自定义Eslint规则。

5. 命令初始化husky：`npx husky install "scripts": {"prepare": "husky install"}`
